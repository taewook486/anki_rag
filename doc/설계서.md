## 1. 프로젝트 개요

### 1.1 문제 정의

Anki는 플래시카드 기반 간격 반복(Spaced Repetition) 학습 도구로, 사용자가 수년간 축적한 24,472개의 영어 학습 데이터(단어, 구동사, 예문)가 `.apkg` 패키지 파일 형태로 존재한다.

그러나 기존 Anki는 다음 한계를 가진다.

- 자연어 질의 불가능: "포기하다라는 뜻의 구동사를 모두 보여줘" 같은 질문에 답하지 못한다.
- 시맨틱 검색 불가능: 의미적으로 유사한 단어 탐색이 불가능하다.
- 맥락 기반 학습 불가능: AI가 컨텍스트를 종합하여 학습 질문에 답변해 주지 않는다.

### 1.2 해결 방안

RAG(Retrieval-Augmented Generation) 아키텍처를 적용하여 Anki 데이터를 벡터 DB에 인덱싱하고, 하이브리드 검색 + LLM 응답 생성을 통해 지능형 영어 학습 도우미를 구현한다.

### 1.3 핵심 사용 시나리오

| 시나리오 | 입력 예시 | 기대 결과 |
|---------|---------|---------|
| 영단어 검색 | "abandon의 뜻은?" | 정의 + 예문 + 발음 정보 반환 |
| 예문 검색 | "포기하다 영어 예문" | 관련 예문 + 출처 반환 |
| 유사어 검색 | "give up 관련 구동사" | 의미적으로 유사한 구동사 목록 |
| 복합 질의 | "TOEFL 경제 관련 학술 단어" | 필터 + 시맨틱 검색 병합 결과 |

### 2.1 Anki 파일 포맷 구조

```
.apkg (ZIP 아카이브)
├── collection.anki21   (신규 버전 SQLite, 우선 사용)
├── collection.anki2    (구버전 SQLite, fallback)
├── media               (JSON: {"0": "aban.mp3", "1": "give_up.mp3", ...})
├── 0                   (바이너리 오디오 파일)
├── 1
└── ...

SQLite notes 테이블
├── id        (note ID)
├── flds      (필드 값, chr(31) = \x1f 로 구분)
└── mid       (model/note-type ID)

SQLite col 테이블
├── models    (JSON: 노트 타입별 필드명 정의)
└── decks     (JSON: 덱 이름 정의)
```

| Basic/forvo 계열 | Front | 뜻 | 발음 | 예문 / 예문 뜻 |
| sentences (txt) | sentence | translation | 없음 | 없음 |

## 4. 핵심 모듈 설계

### 4.1 데이터 모델 (`src/models.py`)

```python
class Document(BaseModel):
    word: str
    meaning: str
    pronunciation: Optional[str] = None
    example: Optional[str] = None
    example_translation: Optional[str] = None
    source: str                          # toefl, xfer, hacker_toeic, ...
    deck: str
    tags: list[str] = []
    note_type: Optional[str] = None
    audio_path: Optional[str] = None     # data/media/{source}/파일.mp3
    difficulty: Optional[str] = None     # 난이도 (beginner/intermediate/advanced)
    synonyms: list[str] = []             # 동의어 목록

class SearchResult(BaseModel):
    document: Document
    score: float                         # RRF fusion score
    rank: int
```

### 4.2 파서 (`src/parser.py`)

**AnkiParser** — .apkg → List[Document]

- ZIP 추출 후 `collection.anki21` 우선 탐색, 없으면 `collection.anki2` fallback
- `notes.flds`를 `\x1f` (chr 31)로 분리하여 필드 추출
- `col.models` JSON으로 노트 타입별 필드명 매핑
- `[sound:파일명]` 정규식으로 오디오 파일 식별 후 `data/media/{source}/`에 추출
- HTML 태그 BeautifulSoup으로 제거

**TextParser** — tab-separated txt → List[Document]

- `utf-8-sig` 인코딩으로 UTF-8 BOM 자동 처리
- 탭 구분, 최소 2개 필드 보장
- 빈 줄 및 불완전 행 자동 스킵

### 4.3 임베더 (`src/embedder.py`)

**BGEEmbedder** — Document → EmbeddingResult

| 항목 | 값 |
|------|-----|
| 모델 | `BAAI/bge-m3` (FlagEmbedding 라이브러리) |
| Dense 벡터 | 1024차원 float32 (cosine similarity) |
| Sparse 벡터 | SPLADE 방식 token weights `Dict[int, float]` |
| 배치 크기 | 기본 32개 |
| 디바이스 | CUDA 자동 감지, 없으면 CPU fallback |
| fp16 | CUDA 환경만 활성화 (`use_fp16 = device != "cpu"`) |
| 문서 임베딩 텍스트 | `{word} {meaning} {example}` 조합 (`embed` / `embed_batch`) |
| 쿼리 임베딩 텍스트 | `"Represent this sentence for searching relevant passages: {query}"` (`embed_query`) |

```python
output = model.encode(
    texts,
    return_dense=True,
    return_sparse=True,
    return_colbert_vecs=False,
)
# dense_vecs[i]: shape (1024,)
# lexical_weights[i]: Dict[token_id, weight]
```

### 4.4 인덱서 (`src/indexer.py`)

**QdrantIndexer** — EmbeddingResult → Qdrant

- Named Vectors 스키마: `dense` (cosine) + `sparse` (SparseIndexParams)
- 페이로드: word, meaning, pronunciation, example, source, deck, tags, audio_path
- `upsert()` — 중복 시 덮어쓰기 (idempotent)

```python
vectors_config = {
    "dense": VectorParams(size=1024, distance=Distance.COSINE),
}
sparse_vectors_config = {
    "sparse": SparseVectorParams(index=SparseIndexParams(on_disk=False)),
}
```

> **QdrantClient 초기화 주의사항**
> `location=` 파라미터는 `:memory:` 또는 HTTP URL 전용이다. 로컬 디렉토리 경로는 반드시 `path=` 파라미터를 사용해야 한다.

```python
# 올바른 초기화
if location == ":memory:" or location.startswith("http"):
    self.client = QdrantClient(location=location)
else:
    self.client = QdrantClient(path=location)  # 로컬 파일 경로
```

### 4.5 리트리버 (`src/retriever.py`)

**HybridRetriever** — query str → List[SearchResult]

RRF (Reciprocal Rank Fusion) 알고리즘:

```
score(doc) = Σ  1 / (rank_i + k)
             i ∈ {dense, sparse}

k = 60  (기본값, 상위 랭크에 과도한 가중치 방지)
```

- `embed_query(query)`: BGE-M3 query instruction prefix 적용 후 임베딩
- `fetch_multiplier=3`: Dense·Sparse 각각 `top_k × 3`개 후보 검색 → RRF 후 `top_k`로 축소 (recall 향상)
- Dense 검색: `query_points(using="dense", query=query_dense_vector)`
- Sparse 검색: `query_points(using="sparse", query=SparseVector(...))`
- 두 결과를 RRF로 병합 후 상위 K개 반환
- `source_filter` + `deck_filter`: Qdrant `FieldCondition` 복합 필터 (`must` AND 조건)

### 4.6 RAG 파이프라인 (`src/rag.py`)

**RAGPipeline** — question str → answer str

```
1. HybridRetriever.search(question, top_k=5)
2. 검색 결과를 구조화된 컨텍스트로 조합
3. 시스템 프롬프트 + 컨텍스트 + 질문을 LLM에 전달
4. 답변 반환 (출처 정보 포함)
```

**LLMProvider 추상화** (교체 가능한 구조):

```
ANTHROPIC_API_KEY 존재? ─Yes─► AnthropicProvider
        │ No
        ▼
LLM_API_KEY 존재? ────────Yes─► OpenAICompatibleProvider
        │ No                     (GLM, OpenAI, OpenRouter 등)
        ▼
ValueError: "LLM_API_KEY 또는 ANTHROPIC_API_KEY 필요"
```

**메시지 구조** (system role 분리 + Few-shot):

```python
messages = [
    {"role": "system", "content": SYSTEM_PROMPT},   # Few-shot 예시 2개 포함
    {"role": "user",   "content": "[참고 자료]\n{context}\n\n[질문]\n{question}"},
]
```

- `AnthropicProvider`: `_extract_system(messages)` 로 system 분리 → `messages.create(system=..., messages=[...])`
- `OpenAICompatibleProvider`: messages 그대로 전달 (OpenAI API가 system role 지원)

**컨텍스트 품질 관리**:

| 상수 | 값 | 설명 |
|------|-----|------|
| `_MIN_RRF_SCORE` | 0.005 | 이 점수 미만 결과 제외 |
| `_MAX_CONTEXT_CHARS` | 2000 | 전체 컨텍스트 최대 문자 수 |
| `_MAX_EXAMPLE_CHARS` | 150 | 예문 필드 최대 문자 수 |

- 컨텍스트 각 항목에 `[순위 N | 점수 X.XXXX]` 메타정보 포함
- `logging.getLogger(__name__)` 사용 (`print()` 제거)

### 4.7 오디오 플레이어 (`src/audio.py`)

플랫폼별 분기:

| OS | 구현 |
|----|------|
| Windows | PowerShell `WMPlayer.OCX` COM 오브젝트 (`playsound` 제거 — Python 3.12+ 호환성 문제) |

## 5. 기능 요구사항

### REQ-001: Anki 패키지 파싱

**WHEN** 사용자가 `.apkg` 파일 경로를 제공하면
**THE SYSTEM SHALL** 해당 파일을 ZIP으로 열고 내부의 SQLite DB를 파싱하여 노트 목록을 반환한다.

- `.anki21` 우선, 없으면 `.anki2` fallback
- `flds`를 `\x1f`(chr 31)로 분리
- `col.models` JSON에서 노트 타입별 필드명 매핑

### REQ-002: 텍스트 파일 파싱

**WHEN** 사용자가 탭 구분 텍스트 파일을 제공하면
**THE SYSTEM SHALL** UTF-8 BOM으로 읽고 탭 분리하여 문장-번역 쌍을 반환한다.

### REQ-003: BGE-M3 임베딩

**WHEN** Document 배치가 준비되면
**THE SYSTEM SHALL** `BAAI/bge-m3` 모델로 Dense(1024차원) + Sparse 벡터를 동시에 생성한다.

### REQ-004: Qdrant 인덱싱

**WHEN** 임베딩이 완료되면
**THE SYSTEM SHALL** Qdrant에 Dense+Sparse 벡터와 메타데이터를 저장한다.

- 컬렉션명: `anki_rag` (설정 가능)
- 페이로드: word, meaning, pronunciation, example, source, deck, tags, audio_path

### REQ-005: 하이브리드 검색 (RRF Fusion)

**WHEN** 사용자가 검색 쿼리를 입력하면
**THE SYSTEM SHALL** Dense + Sparse 검색 결과를 RRF로 병합하여 상위 K개를 반환한다.

- RRF 수식: `score = Σ 1/(rank + 60)`
- 필터 지원: `source`, `deck` 필드

### REQ-006: Claude/OpenAI 호환 RAG 응답 생성

**WHEN** 검색 결과가 준비되면
**THE SYSTEM SHALL** LLM API를 통해 컨텍스트 기반 자연어 답변을 생성한다.

- 우선순위: ANTHROPIC_API_KEY > LLM_API_KEY (OpenAI 호환)
- 출처(source, deck) 정보 포함

### REQ-007: 오디오 파일 추출

**WHEN** `.apkg` 파일을 파싱할 때
**THE SYSTEM SHALL** ZIP 내 미디어 파일을 `data/media/{source}/` 디렉토리에 추출한다.

- `media` JSON: `{"0": "aban.mp3", ...}`
- `[sound:파일명]` 패턴 정규식 추출
- 이미 추출된 파일은 재추출 스킵 (idempotent)

### REQ-008: CLI 인터페이스

```bash
python -m anki_rag index   --data-dir ./data
python -m anki_rag search  "abandon" --source toefl --top-k 5
python -m anki_rag search  "abandon" --play-audio
python -m anki_rag query   "abandon의 뜻과 예문을 알려줘"
python -m anki_rag chat
```

## 12. Agentic AI Agent 구현 방향

### 12.1 설계 배경

기존 RAG 파이프라인은 단일 검색 → 단일 응답의 1-step 구조로, 복잡한 학습 시나리오(여러 단계에 걸친 단어 연관 탐색, 학습 계획 수립, 자기 교정)에는 한계가 있다. Agentic AI Agent 구조를 도입하면 LLM이 검색·응답·평가를 반복적으로 수행하는 멀티스텝 추론이 가능해진다.

### 12.2 Agent 구조 — ReAct 패턴

ReAct(Reasoning + Acting) 패턴을 기반으로 LLM이 도구(Tool)를 선택하고 실행 결과를 관찰하며 다음 행동을 결정한다.

```
[사용자 질문]
     │
     ▼
┌────────────────────────────────────┐
│  Agent Loop (최대 N 스텝)            │
│                                    │
│  Thought: 어떤 도구를 써야 하는가?  │
│     │                              │
│     ▼                              │
│  Action: tool_name(args)           │
│     │                              │
│     ▼                              │
│  Observation: 도구 실행 결과       │
│     │                              │
│     └──► 충분한가? ──No──► 반복    │
│                    Yes             │
└────────────────────────────────────┘
     │
     ▼
[최종 답변 생성]
```

### 12.3 도구(Tool) 정의

| Tool | 함수 시그니처 | 설명 |
|------|-------------|------|
| `search_word` | `(query: str, top_k: int) -> List[Document]` | 하이브리드 벡터 검색 |
| `rag_query` | `(question: str, source: str) -> str` | RAG 기반 자연어 응답 생성 |
| `get_related_words` | `(word: str) -> List[str]` | 유의어·파생어 검색 |
| `filter_by_source` | `(source: str, query: str) -> List[Document]` | 덱/소스 필터 검색 |
| `play_audio` | `(word: str) -> None` | 발음 오디오 재생 |
| `create_study_plan` | `(topic: str, level: str) -> str` | 학습 계획 수립 |

### 12.4 영어 학습 특화 에이전트 시나리오

**시나리오 A — 다단계 어휘 탐색**

```
사용자: "비즈니스 영어에서 '계약 해지'를 표현하는 단어들을
         난이도 순서로 정리해줘"

Step 1 - Thought: source=toefl, xfer로 관련 단어 검색 필요
Step 1 - Action: search_word("contract termination", source="toefl")
Step 1 - Observation: [cancel, terminate, void, rescind, ...]

Step 2 - Thought: phrasal 덱에서 구동사도 보완 필요
Step 2 - Action: search_word("cancel agreement", source="phrasal")
Step 2 - Observation: [call off, back out, pull out, ...]

Step 3 - Thought: 결과를 종합하여 난이도 순 정렬 후 응답
Step 3 - Action: rag_query("위 단어들을 난이도 순으로 설명")
Final Answer: cancel(기초) → terminate(중급) → rescind(고급) ...
```

**시나리오 B — 자기 교정(Self-Correction)**

검색 결과의 RRF 점수가 `_MIN_RRF_SCORE(0.005)` 미만이면 에이전트가 쿼리를 재작성하여 재검색을 시도한다.

```
Action: search_word("nullify")
Observation: score=0.003 (임계값 미만, 결과 없음)

Thought: 동의어로 재시도 필요
Action: search_word("invalidate contract")
Observation: score=0.041 → 정상 결과 반환
```

### 12.5 구현 방향

- **단기 (v1.2)**: FastAPI에 `/api/agent` 엔드포인트 추가, Tool 함수 래핑, 최대 5스텝 ReAct 루프 구현
- **중기 (v1.3)**: 학습 이력 저장소(SQLite) 연동, 개인화 학습 계획 생성 에이전트
- **장기 (v2.0)**: 멀티에이전트 구조 — 검색 에이전트·평가 에이전트·학습 플래너 에이전트 분리

```python
# Agent 진입점 (구현 예정)
class LearningAgent:
    def __init__(self, retriever: HybridRetriever, rag: RAGPipeline):
        self.tools = {
            "search_word": retriever.search,
            "rag_query": rag.query,
        }

    def run(self, question: str, max_steps: int = 5) -> str:
        history = []
        for _ in range(max_steps):
            action = self._think(question, history)
            if action.is_final:
                return action.answer
            observation = self.tools[action.tool](**action.args)
            history.append((action, observation))
        return self._summarize(history)
```

## 13. 최신 RAG 트렌드 및 지식 그래프 연계 설계

### 13.1 최신 RAG 패러다임 비교

| 방식 | 특징 | 본 프로젝트 적용 여부 |
|------|------|-------------------|
| Naive RAG | 단순 검색 → 생성 | ✅ 기본 구현 완료 |
| Advanced RAG | Hybrid 검색, Re-ranking, 쿼리 재작성 | ✅ BGE-M3 Hybrid + RRF |
| Modular RAG | 파이프라인 구성요소 교체 가능 | ✅ LLMProvider 추상화 |
| GraphRAG | 지식 그래프 기반 커뮤니티 요약 검색 | 🔜 v2.0 예정 |
| Self-RAG | 검색 필요 여부 자율 판단, 비판적 자기 평가 | 🔜 v1.2 Agent에 통합 예정 |
| Adaptive RAG | 쿼리 복잡도에 따라 검색 전략 동적 조정 | 🔜 v1.3 예정 |
| Corrective RAG | 검색 결과 품질 자가 평가 후 재검색 | 🔜 v1.2 Agent에 통합 예정 |

### 13.2 지식 그래프 설계

영어 단어 간 의미 관계를 그래프로 구조화하여 벡터 검색의 한계(문맥 단절)를 보완한다.

**노드(Node) 유형**

| 노드 타입 | 예시 | 설명 |
|----------|------|------|
| `Word` | abandon | 단어 엔티티 |
| `Meaning` | 포기하다 | 의미 엔티티 |
| `Category` | TOEFL, Business | 도메인 분류 |

**엣지(Edge) 유형**

| 관계 | 예시 | 설명 |
|------|------|------|
| `SYNONYM` | abandon → give up | 유의어 관계 |
| `ANTONYM` | accept ↔ reject | 반의어 관계 |
| `DERIVED_FROM` | abandonment ← abandon | 파생어 관계 |
| `CO_OCCURS` | abandon — project | 공기 관계 (예문 기반) |
| `SAME_CATEGORY` | abandon — quit | 동일 도메인 묶음 |

**그래프 데이터 흐름**

```
AnkiParser 파싱 결과
        │
        ▼
  [graph_builder.py]
  (노드: Word, 엣지: SYNONYM/ANTONYM/DERIVED_FROM)
        │
        ▼
  Neo4j 또는 NetworkX 그래프 저장
        │
        ├── Qdrant 벡터 검색 결과
        │          │
        └──────────▼
          GraphRAG Fusion
          (벡터 유사도 + 그래프 연결성 결합)
                  │
                  ▼
              최종 컨텍스트
```

### 13.3 Self-RAG 적용 설계

Self-RAG는 LLM이 검색 필요 여부(Retrieval token), 결과 관련성(ISREL), 응답 지지도(ISSUP), 유용성(ISUSE)을 스스로 평가하는 방식이다.

본 프로젝트에서는 다음과 같이 단순화하여 적용한다.

```
사용자 질문 입력
      │
      ▼
[검색 필요 여부 판단]
 LLM: "이 질문은 Anki 데이터를 검색해야 하는가?"
  ├── Yes → HybridRetriever.search() 실행
  └── No  → LLM 자체 지식으로 응답 (인사, 문법 규칙 등)
      │
      ▼
[검색 결과 품질 평가]
 LLM: "검색 결과가 질문에 충분히 관련이 있는가?"
  ├── Yes (score ≥ 임계값) → 컨텍스트로 사용
  └── No  → 쿼리 재작성 후 재검색 (최대 2회)
      │
      ▼
[최종 응답 생성 및 출처 검증]
```

### 13.4 Adaptive RAG 적용 설계

쿼리의 복잡도를 분류하여 검색 전략을 동적으로 결정한다.

| 쿼리 복잡도 | 판단 기준 | 검색 전략 |
|------------|---------|---------|
| 단순 (Simple) | 단일 단어 검색 ("abandon") | Dense 검색만, top_k=5 |
| 중간 (Moderate) | 의미 관계 질의 ("give up 유의어") | Hybrid 검색, top_k=10, RRF |
| 복잡 (Complex) | 다단계 추론 ("TOEFL 경제 단어를 난이도별로") | Agent Loop + 지식 그래프 병행 |
